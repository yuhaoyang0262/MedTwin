# MedTwin：医患双端乳腺癌风险预测与病例管理系统

MedTwin 是一个面向数据结构课程大作业的医患双端病例管理与乳腺癌风险预测演示系统。系统围绕“病例录入、风险预测、医生复核、病例检索、排序管理、队列处理、撤销操作”等业务流程，将顺序表、队列、栈、查找、排序和字典映射等数据结构知识融入到可运行的 Web 项目中。

> 注意：本系统仅用于课程演示和辅助筛查，不替代临床诊断。预测结果不能作为疾病诊断、治疗决策或医疗处置依据。

## 项目背景

乳腺癌风险评估通常需要综合病史、生活方式、家族史、代谢指标等多类信息。传统表格化管理方式容易出现信息分散、病例复核流程不清晰、查询和排序效率较低等问题。

MedTwin 以“患者端录入 + 医生端管理”为核心场景，构建了一个轻量级病例管理系统：

- 患者端负责填写基础信息、病史因素和生活方式指标，并触发风险预测。
- 医生端负责查看病例列表、处理待复核队列、按条件检索病例、按风险或时间排序，并支持撤销操作。
- 系统通过已有训练模型输出风险概率、风险等级和主要影响因素，用于课程场景下的辅助展示。

## 系统功能

### 患者端

- 按“基础信息、病史因素、生活方式”分组录入病例指标。
- 支持“只预测”，不写入病例库，仅查看当前输入对应的风险结果。
- 支持“预测并入库”，将病例加入病例列表，供医生端查看和复核。
- 输出风险概率、风险等级、简短进度条和主要影响因素标签。

### 医生端

- 查看全部病例列表。
- 按患者姓名或病例编号检索病例。
- 按风险、时间、年龄、姓名排序病例。
- 查看每个病例的风险概率、风险等级和主要影响因素。
- 处理待复核队列中的队首病例。
- 支持撤销最近一次入库、队列处理等操作。

### 首页展示

- 展示系统名称、双端业务定位和当前运行状态。
- 展示模型状态、病例总数、待复核数量和当前阈值。
- 展示数据结构课程知识点与系统功能的对应关系。

## 技术路线

系统采用前后端轻量化实现方式，重点服务课程演示和算法结构展示。

- 前端：HTML、CSS、JavaScript
- 后端：Python 标准库 HTTP 服务
- 模型：已训练好的 XGBoost 风险预测模型
- 数据管理：内存数据结构保存病例、队列和撤销状态
- 部署展示：支持本地 Python 后端运行，也支持 GitHub Pages 静态演示版本

整体流程如下：

1. 用户在患者端填写病例指标。
2. 前端将指标组织为结构化特征数据。
3. 系统调用模型预测逻辑，得到风险概率。
4. 系统根据概率生成风险等级和主要影响因素。
5. 如果选择入库，病例进入病例顺序表；高风险病例进入医生复核队列。
6. 医生端对病例执行查找、排序、队列处理和撤销操作。

## 医患双端设计

MedTwin 将业务流程拆分为患者端和医生端，体现医患双端分工。

### 患者端职责

- 完成原始指标录入。
- 发起风险预测。
- 将病例提交给系统管理。

### 医生端职责

- 查看患者端提交的病例。
- 重点关注高风险或待复核病例。
- 根据队列顺序处理复核任务。
- 使用查询、排序和撤销功能管理病例数据。

这种设计使系统流程更接近真实医疗管理场景，也便于在答辩中展示不同数据结构在不同业务环节中的作用。

## 机器学习模型说明

系统调用已有训练好的 XGBoost 模型进行乳腺癌风险预测。模型输入来自患者端表单中的结构化指标，包括：

- 基础信息：学历、医保类型等。
- 病史因素：乳腺癌家族史、良性乳腺病史、恶性肿瘤家族史、代谢综合征、二型糖尿病、激素替代疗法等。
- 生活方式：BMI 分类、腰臀比、静态行为时间、中高强度运动、睡眠时长、抑郁得分等。

模型输出包括：

- 风险概率
- 风险等级
- 主要影响因素
- 当前模型状态和阈值

本项目中的模型用于课程演示和辅助筛查。由于医疗风险预测涉及复杂临床因素，系统输出不能替代医生诊断。

## 数据结构应用说明

本项目的重点是将数据结构课程知识应用到一个完整业务系统中。主要对应关系如下。

| 数据结构或算法 | 系统实现位置 | 具体作用 |
| --- | --- | --- |
| 顺序表 | `store.cases` | 用于存储病例列表。病例按照数组形式保存，支持遍历、追加、筛选和排序。 |
| 队列 | `store.queue` | 用于医生复核队列。高风险或待复核病例按入队顺序等待处理，医生端通过“处理队首”完成出队。 |
| 栈 | `store.undo_stack` | 用于撤销操作。每次入库、队列处理等关键操作前保存快照，撤销时后进先出恢复上一步状态。 |
| 排序算法 | 病例列表排序逻辑 | 用于病例按风险、时间、年龄、姓名排序，帮助医生快速定位重点病例。 |
| 查找算法 | 病例检索逻辑 | 用于按患者姓名或病例编号检索病例，实现医生端快速查询。 |
| 字典/映射 | 字段映射、特征映射、模型输入组织 | 用于将前端字段映射到模型特征名称，组织模型输入，并维护影响因素标签。 |
| 哈希表 | 医生端病例索引 | 使用 `Map` 构建病例编号和患者姓名索引，支持按编号或姓名快速定位病例。 |
| 树形结构 | 风险分层树 | 将病例按照极高风险、高风险、中风险、低风险进行层级化组织，展示不同风险层级的病例数量。 |

### 顺序表：病例列表 `store.cases`

病例数据以列表形式保存。每个病例包含编号、姓名、年龄、来源端口、状态、特征数据和预测结果。医生端渲染病例表格时会遍历该顺序表，并根据查询条件和排序方式生成当前视图。

### 队列：医生复核队列 `store.queue`

当病例需要医生复核时，病例编号进入队列。医生点击“处理队首”时，系统取出队首元素并更新病例状态，体现先进先出的队列特点。

### 栈：撤销操作 `store.undo_stack`

在执行会改变系统状态的操作前，系统将当前病例列表、复核队列等状态压入撤销栈。用户点击“撤销”时，系统弹出栈顶状态并恢复，体现后进先出的栈特点。

### 排序算法

医生端可按不同字段对病例列表排序：

- 按风险排序：优先查看风险概率更高的病例。
- 按时间排序：查看最新录入病例。
- 按年龄排序：按患者年龄组织病例。
- 按姓名排序：便于按姓名浏览。

### 查找算法

医生端搜索框支持按患者姓名或病例编号查询。系统遍历病例列表，对姓名和编号进行匹配，返回符合条件的病例。

### 哈希表：病例快速索引

医生端在加载病例列表时，会根据当前病例顺序表构建哈希索引。系统分别以病例编号和患者姓名作为键，将对应病例对象保存到 `Map` 中。当用户输入完整病例编号或姓名时，可以通过哈希表快速定位病例；当输入的是模糊关键词时，再退回到列表匹配。该设计体现了哈希表在快速查找场景中的应用。

### 树形结构：风险分层树

系统根据模型输出的风险等级构建风险分层树。根节点表示全部病例，下一层按照极高风险、高风险、中风险、低风险划分，每个子节点记录该风险等级下的病例数量。医生端通过该结构可以直观看到病例在不同风险层级中的分布情况，便于优先关注高风险人群。

### 字典/映射

系统中使用字典维护字段映射和模型特征映射，例如前端字段名与模型中文特征名之间的对应关系。该设计可以降低硬编码复杂度，使模型输入组织更加清晰。

## 本地运行方法

建议使用已安装 XGBoost、pandas、scikit-learn、joblib 等依赖的 Python 环境。

## 答辩演示流程

建议答辩时按照以下顺序展示。

1. 打开首页，介绍项目名称、应用背景和医患双端设计。
2. 展示首页运行状态，包括模型状态、病例总数、待复核数量和阈值。
3. 说明“课程知识点对应关系”，指出顺序表、队列、栈、排序、查找和映射在系统中的位置。
4. 进入患者端，填写一组低风险病例，点击“只预测”，展示风险概率和影响因素。
5. 修改为包含家族史、良性乳腺病史、代谢因素或生活方式风险的病例，再次点击“只预测”，观察风险变化。
6. 点击“预测并入库”，展示病例进入医生端病例列表。
7. 进入医生端，演示按风险排序、按姓名或编号查询。
8. 演示待复核队列，点击“处理队首”，说明队列先进先出。
9. 点击“撤销”，说明栈的后进先出特点。
10. 总结项目如何将数据结构知识与机器学习预测系统结合。

## 注意事项

- 系统仅用于课程演示和辅助筛查，不替代临床诊断。
- 模型结果不能作为医学诊断、治疗方案或临床决策依据。
- 当前病例数据主要保存在运行时内存或浏览器本地状态中，适合作业展示，不等同于生产级数据库系统。
- 如果部署在 GitHub Pages，预测逻辑会在浏览器端运行；如果本地运行 `app.py`，预测接口由 Python 后端提供。
